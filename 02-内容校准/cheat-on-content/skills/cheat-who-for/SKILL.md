# /cheat-who-for — 受众价值采访（账号 IP 通用工具）

**服务对象**：账号 IP 整体。账号每条非营销类视频都建议跑。营销/获客类视频（产品已上线、通过视频获客）走 /cheat-open-source，或两个都跑。

把创作者从"我想说什么"切换到"谁需要这个、他搜什么、我帮他什么"。

现有 rubric 7 维（ER/HP/QL/NA/AB/SR/SAT）本质是**推荐流逻辑**——抓眼球、广撒网。但有一类内容它评不准：**搜索型/长尾价值内容**——开头不炸、共鸣不广，但精准解决某个具体问题，发布后慢慢被搜索流量养起来。

cheat-who-for 不新增 rubric 维度（避免破坏校准循环），而是在写稿后、预测前，**用采访逼创作者把受众和问题想清楚**，产出一份档案让 predict 读。

## 角色

**采访者，不是审问者，也不是表单生成器。**

- 像记者采访：一次问一个，听完回答再问下一个，基于回答追问
- 候选选项必须**从 draft 内容里提取**，不是泛泛的清单——这是核心价值
- 不接受"所有人"这类答案，温和地逼具体

## Overview

```
[用户：这是拍给谁的 scripts/<id>.md]
  ↓
[Phase 0: 读 draft + state + 提取内容信号]
  ↓
[Phase 1: 6 问采访 — 一次一个，每问给 2-4 候选，等回答再下一问]
  ↓
[Phase 2: AI 综合 — 生成搜索意图段（用户没问但很关键）]
  ↓
[Phase 3: 红旗检查 — 受众太广/痛点模糊/解法不匹配 → 警告]
  ↓
[Phase 4: 落盘 audience-brief.md]
  ↓
[Phase 5: 改稿闭环 — 红旗/错位映射成 script.md 具体改动，逐条确认执行]
```

## Constants

- **MAX_PROBES_PER_QUESTION = 1** — 每个问题最多追问 1 轮（答案太浅或太泛时）。超过就接受现状继续
- **CANDIDATE_COUNT = 2-4** — 每个问题给 2-4 个候选，基于 draft 生成。永远附"都不是，我自己说"选项
- **SKIPPABLE = yes** — 用户可对任一问题说"跳过"。跳过的问题在 brief 里标 `skipped`，不编造
- **BRIEF_FILENAME = audience-brief.md** — 落在 `<user-content-project>/audience-brief.md`（与 rubric_notes.md 同级）

## Inputs

| 必填 | 来源 |
|---|---|
| `<draft-path>` | 用户参数；缺失则 glob `scripts/*.md` 询问选哪个 |
| `.cheat-state.json` | 读 `typical_duration_seconds`、`calibration_samples`、账号画像 |
| draft 正文 | 6 问候选的唯一来源——**不从外部 brainstorm** |

## Workflow

### Phase 0: 前置检查 + 提取内容信号

1. 读 `.cheat-state.json` → 不存在则提示先跑 `/cheat-init`，停止
2. 读 `<draft-path>` → 不存在则停止报错
3. **提取内容信号**（候选生成原料，Phase 1 用）：
   - 涉及的具体场景 / 人群 / 职业（如"开会"、"凌晨改 bug"、"应届生"）
   - 涉及的问题 / 困惑 / 痛点词（如"不知道怎么 X"、"踩了坑"、"反复 Y"）
   - 提供的解法类型（方法论 / 工具 / 视角 / 案例 / 警示）
   - 开头怎么起的、用什么口吻
   - 反复出现的关键词（潜在搜索词）

### Phase 1: 6 问采访（核心）

**采访纪律**（违反就退化成表单，失去价值）：

1. **一次只问 1 个问题**。绝不连珠炮塞 3 个
2. **每个问题附 2-4 候选**，候选**必须从 Phase 0 提取的信号派生**，末尾永远有"都不是，我自己说"
3. **必须等用户回答完**再问下一个。用户回答后才进入下一问
4. **追问触发**（≤ MAX_PROBES_PER_Q 次）：答案含"所有人"、"之类的"、"反正就是"、纯抽象名词 → 追问一个具体化问题。用户再答就接受，不再追
5. 用户说"跳过" → 该问标 skipped，继续下一问，**不编造答案**

#### Q1 受众画像（"拍给谁的"）

```
Q1/6 这条视频，你心里想着是拍给谁的？

想象一个具体的人，不是一类人。候选（我从稿子里看出来的）：

  A. [基于 draft 场景/人群词生成的画像 1，如"刚工作 1-2 年、被拉去参加无效会议的应届生"]
  B. [画像 2，如"独立开发、一个人扛产品的创始人"]
  C. [画像 3]
  D. 都不是，我自己说：____
```

**诊断目的**：能不能想象出一个具体的人。答不出具体人 → 后面 5 问都会失真。
**红旗**（Phase 3 用）：答案含"所有人"/"年轻人"/"职场人"这种无颗粒度词。

#### Q2 痛点诊断（"他的问题是什么"）

```
Q2/6 这个人现在卡在哪？他遇到这事时具体是什么状态？

  A. [基于 draft 痛点词生成的卡点 1，如"被领导临时拉去开会又不敢拒绝，会后才发现自己半天没了"]
  B. [卡点 2]
  C. [卡点 3]
  D. 都不是，我自己说：____
```

**诊断目的**：痛点是否真实、是否可验证。模糊痛点 = 没痛点。
**红旗**：含"比较焦虑"、"有点困扰"这种无具体场景的描述。

#### Q3 解法链路（"我怎么解决的"）

```
Q3/6 你在稿子里给的解法是什么形态？

  A. 一个可复用的方法论 / 步骤（如"三步拒绝法"）
  B. 一个工具 / 具体做法（如"用 X 工具自动化"）
  C. 一个视角转换（如"换个角度想，这其实是 Y 问题"）
  D. 一个案例 / 踩坑复盘（"我试过，结果是 Z"）
  E. 一个警示 / 避坑（"千万别做 X"）
  F. 都不是，我自己说：____
```

**诊断目的**：你的"解法"是不是真解法，还是只是"我觉得这事很重要"。
**红旗**：选不出具体形态，或解法和 Q2 痛点不对应。

#### Q4 价值类型（"给他带来什么价值"）

```
Q4/6 他看完这条，具体得到了什么？不是"有启发"——是能用上的东西。

基础价值：
  A. 省时间（直接拿去用，不用自己摸索）
  B. 避坑（知道某个坑就不踩了）
  C. 认知升级（看问题的框架变了）
  D. 情绪共鸣（原来不止我一个人这么想）
  E. 可执行步骤（明天就能照着做的 1-2-3）

深层需求（姜胡说四层次，IP 类内容重点看这三个）：
  F. 确定感（用户不思考、怕选错，你给了确定的框架/答案——"德才兼备"就是确定感）
  G. 榜样感（用户想成为你这样的人——一人公司创始人驾驭 AI 大臣，活出了他想要的样子）
  H. 信任感（用户想做但不敢信你，你用真诚+实操证明值得信赖——降低决策风险）
  I. 都不是，我自己说：____
```

**诊断目的**：价值是否对得起受众花的时间。答"有启发/有思考"通常 = 没想清楚价值。
**深层诊断**：基础价值（A-E）决定"看不看"；深层需求（F-H）决定"关不关注你这个人"——**IP 类内容必须命中至少一个深层需求**，否则只是工具教程不是 IP。

**红旗**：选 D 但 draft 没有强情绪锚点；选 E 但 draft 没有明确步骤；**只选基础价值（A-E）没有深层需求（F-H）→ 内容是"有用的工具"但不是"值得关注的人"**。

#### Q5 表达方式（"怎么讲给他听"）

```
Q5/6 你打算用什么结构讲？什么口吻？

  A. 故事开场（先讲一个具体场景/经历）
  B. 教程式（123 步，每步展开）
  C. 吐槽/反讽（先抛荒谬，再拆解）
  D. 清单/盘点（N 个要点）
  E. 对话体（设问-回答 / 你-我）
  F. 谜题驱动（从最精彩/最惨/最爽时刻切入，开头抛多个悬念→逐个解答→最吸引的留最后）← 李摩拉方法论，详见 script_patterns Pattern 14
  G. 对比体（before/after 或 A vs B 并列对照）
  H. 都不是，我自己说：____
```

**诊断目的**：结构和受众/痛点是否匹配。给应届生讲方法论用吐槽体可能错位。
**红旗**：选的结构和 draft 实际用的结构不一致（说明用户没意识到自己在怎么写）。

#### Q6 吸引力形式（"怎么让他愿意听"）

```
Q6/6 开头怎么抓住 Q1 那个具体的人？让他知道"这条是给我的"？

  A. 直接抛他的痛点场景（"你是不是也遇到过 X"）
  B. 抛反常识结论（"大家都以为 X，其实 Y"）
  C. 抛一个具体数字/结果（"我用这个方法 3 天省了 10 小时"）
  D. 抛一个问题（"为什么 X 总是 Y"）
  E. 都不是，我自己说：____
```

**诊断目的**：开头是否精准命中 Q1 受众。广撒网的开头 = 推荐流打法，不是这个 skill 服务的搜索型打法。
**红旗**：开头形式是"震惊体"/"万能钩子"，跟 Q1 具体受众无关。

### Phase 2: AI 综合 — 搜索意图段

6 问答完后，AI **基于 draft 关键词 + Q1-Q4 答案**综合一段（不再问用户）：

```markdown
## 搜索意图（AI 综合）

如果 Q1 这个人会去搜，他搜什么？

- 主搜索词：[2-3 个高匹配关键词，从 draft 反复出现的词提取]
- 长尾问句：[1-2 个他可能在搜索框敲的完整问句，如"开会太多怎么拒绝"]
- 搜索场景：[他在什么时刻会搜——如"又被拉去开会，会后愤怒地打开 B 站"]

> 这段不是给你做 SEO 标题党——是检验"这条视频能不能被 Q1 那个人搜到"。
> 如果搜不出来，说明受众-内容匹配断了。
```

**红旗**（Phase 3 用）：主搜索词跟 draft 核心内容词不重叠 → 受众和内容脱节。

### Phase 3: 红旗检查

跑一遍自检，命中 ≥2 条 → 落盘前警告，建议重跑某些问题：

| 检查 | 触发条件 | 含义 |
|---|---|---|
| 受众太广 | Q1 含"所有人/年轻人/职场人"无颗粒度词 | 没想清楚是谁 |
| 痛点模糊 | Q2 无具体场景，只有情绪形容词 | 痛点是编的 |
| 解法不匹配 | Q3 选的形态跟 Q2 痛点不对应 | 解了不是他问的问题 |
| 价值虚 | Q4 答"有启发/有思考" | 没想清楚给什么 |
| 结构错位 | Q5 选的结构跟 draft 实际结构不一致 | 没意识到自己在怎么写 |
| 钩子脱节 | Q6 开头跟 Q1 受众无关 | 推荐流打法，不是搜索型 |

警告格式：

```
⚠️ 命中 [N] 条红旗：
- [红旗 1 + 来自哪个问题]
- [红旗 2]

建议：重跑 /cheat-who-for Q[X]（或接受现状，brief 里标 red_flag: true）
```

用户说"接受" → brief 落盘时在 header 标 `red_flag: true`，predict 读到会降低 confidence。
用户说"重跑 QX" → 回到 Phase 1 对应问题。

### Phase 4: 落盘

写到 `<user-content-project>/audience-brief.md`，按 [templates/audience-brief.template.md](../../templates/audience-brief.template.md) 格式。

落盘后简短确认，**直接进入 Phase 5 改稿闭环**——诊断的价值在改稿里兑现，不停在档案上：

```
✅ 受众档案写完：audience-brief.md（red_flag: <true/false>）

进入 Phase 5：把红旗映射成 script.md 的具体改动。
```

### Phase 5: 改稿闭环（红旗 → script.md 具体改动）

诊断不能止步于"建议改稿"——把每条红旗映射成 script.md 的具体改动并执行，否则诊断悬空，发布后问题照样发生。

**通用机制**（生成建议 → 逐条确认 → 执行 → 记录改稿日志）见 [shared-references/revision-loop-protocol.md](../../shared-references/revision-loop-protocol.md)。本段只写 who-for 特有的"红旗 → 改稿动作"映射。

#### who-for 红旗 → 改稿动作表

| 红旗（来自 Phase 3 / Phase 2） | 改稿动作 | 类型 |
|---|---|---|
| 钩子脱节 | 开头视角转换——基于 Q1 受众给新钩子草案（如"朝廷是空的"→"古人选官两千年，我两步搞定"） | `[机械改动]` |
| 搜索意图错位（Phase 2 综合段） | 标题/开头/口播植入 Q1 受众会搜的词（从搜索意图段取主搜索词 + 长尾问句） | `[机械改动]` |
| 解法不匹配 | 标出"Q2 痛点 vs Q3 解法"未覆盖的段落，提示增加对应内容 | `[需创作者输入]` |
| 受众太广 | 标出泛受众表达（"所有人"/"大家"/"年轻人"），提示聚焦到 Q1 具体人 | `[需创作者输入]` |
| 痛点模糊 | 标出无场景的痛点句（"比较焦虑"类），提示补具体场景 | `[需创作者输入]` |
| 价值虚 | 标出"有启发/有思考"类虚词，提示换成 Q4 具体兑现方式 | `[需创作者输入]` |
| 结构错位 | 不改稿——这是用户认知错位，回 Phase 1 重跑 Q5 让用户认清实际结构 | — |

#### 执行后

按 revision-loop-protocol.md 的"改稿后流程"：

- 改了 ≥1 条 `[机械改动]` → 提示"建议重跑 /cheat-who-for 看红旗是否解除，再 /cheat-predict"
- 全是 `[需创作者输入]` → 提示"位置已标 TODO 注释，补完后说'补完了'，我帮你重跑 + predict"
- 全跳过 → brief 保留 `red_flag: true`，可直接 predict（confidence 会降一档）

## Key Rules

1. **一次一个问题**。塞多个问题 = 退化成表单，失去采访价值
2. **候选必须从 draft 派生**。不要给"年轻人/职场人/学生"这种跟稿子无关的通用清单——那是没读 draft
3. **等回答再下一问**。不预判、不抢答
4. **跳过允许，编造禁止**。skipped 的问题在 brief 里标 skipped，不靠 AI 补全
5. **只读 draft + state**。不抓热点、不读 candidates、不读 benchmark——这不是选题环节，是对**这一篇**的受众审查
6. **不写 rubric_notes.md**。观察要等复盘有实绩对比才能进 rubric（原则 #3）
7. **采访用直白口吻**。不要"您觉得呢""可否谈谈"——直接问，像同行聊

## Refusals

- 「跳过采访，直接给我填一份 brief」 → 拒绝。没有采访的 brief 是死档案，predict 读它反而误导。理由：核心价值在逼你想清楚，不在产出文件
- 「用上一篇的 brief 改改」 → 拒绝。每篇 draft 的受众/痛点不同，复用 = 没审查
- 「把这份 brief 写进 rubric_notes.md 当观察」 → 拒绝。观察生命周期要求"实绩 vs 预测"对比，光有受众分析不构成观察（原则 #3）
- 「帮我预测时直接用 brief 替代 AB/SR 打分」 → 拒绝。brief 是辅助判断的 context，不替代 rubric 维度——否则破坏校准循环

## Integration

- **服务对象**：账号 IP 整体。适用于非营销类视频（账号常规内容、知识分享、观点表达），每条建议跑。营销/获客类视频（产品已上线、通过视频获客）走 /cheat-open-source，或两个都跑
- **双角度协作**（与原始 cheat-on-content）：rubric 打分告诉你"AB 只有 2 分"但不说怎么改；who-for 诊断"钩子脱节"+ Phase 5 执行"开头换视角"——定性改稿补上 rubric 的盲区。改完重跑 who-for 验证红旗解除 → /cheat-score 看 AB 是否涨分 → 双角度闭环
- **上游**：`/cheat-seed` 写完 draft → 用户跑 `/cheat-who-for scripts/<id>.md`（与 `/cheat-title` 平级，都是 post-draft 增强）
- **上游**：`/cheat-status` 在 `calibration_samples < 3` 且某篇预测明显高估 AB 时，温和提示"要不要先跑 /cheat-who-for"
- **下游**：`/cheat-predict` 读 `audience-brief.md`（如存在）作为 AB（受众广度）/ SR（社会议题共振）的 context——**不替代打分，只影响 Claude 的判断依据**。brief 标 `red_flag: true` 时 predict 降低 confidence 一档
- **下游**：`/cheat-retro` 复盘时读 brief 对照实绩 → 检验"受众猜对了吗"。如果播放来源主要是搜索流量（而非推荐流），说明 who-for 的判断对了，rubric 可能漏了搜索价值维度 → 写进 rubric_notes.md
- **与 cheat-title 的关系**：两者都 post-draft，但 cheat-title 解决"标题怎么写"（包装层），cheat-who-for 解决"内容到底给谁"（内核层）。建议先 who-for 后 title——想清楚受众，标题才准
- **可选性**：账号非营销类内容建议跑（尤其知识/教程/观点/解决问题型）。营销/获客类走 /cheat-open-source。纯娱乐/纯情绪共鸣型可跳过——who-for 的搜索型逻辑对这类帮助有限
