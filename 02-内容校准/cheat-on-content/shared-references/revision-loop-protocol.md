# 改稿闭环协议（Revision Loop Protocol）

> 被 `/cheat-who-for` 和 `/cheat-open-source` 的 Phase 5 共同引用。
> 这两个 skill 的诊断产出（audience-brief.md / open-source-audit.md）不能止步于"建议改稿"——
> 必须把红旗/错位映射成 script.md 的具体改动并执行，否则诊断悬空，无法影响流量。
>
> 本协议定义通用的改稿机制（生成建议 → 确认 → 执行 → 记录）。
> 各 skill 的 Phase 5 只需写自己的"红旗 → 改稿动作"映射表，机制复用本协议。

## 核心理念

**诊断的价值在改稿里兑现，不在档案里。**

一份标了 red_flag: true 的 audience-brief，如果稿子没改，发布后问题照样发生。Phase 5 的存在就是为了堵这个缺口——让每一条红旗都尝试落到 script.md 的具体改动上。

## 与原始 cheat-on-content 的协作（双角度）

```
角度 1（原始 cheat-on-content）：量化打分 → 预测 → 复盘 → 进化 rubric
   ↓ 告诉你"这条预计多少播放、哪个维度拖后腿"
角度 2（who-for + open-source）：定性诊断 → 改稿 → 重跑验证
   ↓ 告诉你"具体哪段怎么改，能修复那个拖后腿的维度"

两者共同指向：提高流量 / 内容质量
```

rubric 打分发现"AB（受众广度）只有 2 分"，但**不说怎么改**。
who-for 诊断"钩子脱节"+ Phase 5 改稿"开头从 A 视角换成 B 视角"——这是 rubric 给不了的。
改完重跑 who-for 看红旗是否解除 → 重跑 cheat-score 看 AB 是否涨分 → 双角度闭环。

## 触发条件

Phase 5 在 Phase 4（落盘 brief/audit）之后自动进入，**无论 red_flag 是否为 true**：
- red_flag: true → 改稿建议聚焦于命中的红旗
- red_flag: false → 仍生成"增强建议"（基于 Phase 2 的综合段，如搜索意图可优化处），但标为 optional

用户可随时说"跳过改稿"，Phase 5 结束。

## 建议条目格式

每条改稿建议必须包含 5 个字段：

```markdown
### 改稿建议 #[N]

- **位置**: script.md 的具体段落/行（引用原文一句，便于定位）
- **问题**: 来自哪条红旗 / 哪个 Phase 2 发现（如"钩子脱节：开头是开发者视角"）
- **改法**: 一句话说清怎么改（如"开头视角从'朝廷是空的'换成历史分量+反差"）
- **草案**: AI 给的具体改写文本（用户可改）
- **类型**: `[机械改动]` | `[需创作者输入]`
```

### 类型判定标准

**`[机械改动]`** — AI 可直接执行（用户确认后）：
- 视角转换（开头钩子换形式）
- 删营销话术 / 去套路化表达
- 改口吻（"应该/记住" → "我当时/我发现"）
- 关键词植入（标题/开头/标签加搜索词）
- 段落顺序调整
- 删冗余铺垫

**`[需创作者输入]`** — AI 只给提示和位置，**不替写**（因为涉及个人经历/推理过程/语气）：
- 补推理过程（open-source Q3 缺失）
- 补真实经历锚点（open-source Q1 经验地基）
- 补卡点场景（open-source Q2）
- 加个人差异化印记（open-source Q5）
- 内容比例大改（who-for 受众错位时的增删）

> 红线：`[需创作者输入]` 的条目，AI **绝不**替写草案。只给"这里缺什么、大概多少字、参考方向"，等用户自己写完再继续。违反 = 把 cheat-seed 的"AI 给脚手架，你必须改写"原则打破了。

## 确认机制（逐条）

每条建议单独确认，**不批量执行**：

```
改稿建议 #1：
  位置：[原文引用]
  问题：[红旗来源]
  改法：[一句话]
  草案：[AI 改写文本]  ← [机械改动] 才有这一行
  类型：[机械改动] / [需创作者输入]

你的选择：
  [1] ok，执行（机械改动 → AI Edit；需创作者输入 → 标记位置等你自己写）
  [2] 草案不对，我改一版：（用户给新草案）→ AI 用新草案执行
  [3] 跳过这条
  [4] 这条我自己改，不用 AI 动
```

用户选 [1] 机械改动 → AI 用 Edit 工具改 script.md
用户选 [1] 需创作者输入 → AI 在 script.md 对应位置插入提示注释 `<!-- TODO(需创作者输入): 补推理过程——你怎么从X得出Y的，~80字 -->`，不替写
用户选 [2] → 用用户的新草案执行 Edit
用户选 [3] → 跳过，记录"用户拒绝"
用户选 [4] → 不动 script.md，记录"用户自改"

## 执行规则

1. **只改用户确认的条目**。未确认的不动
2. **每次 Edit 后即时反馈**："✅ 改了第 X 段：'[原文]' → '[新文]'"
3. **保留原文风格**：草案匹配稿子的现有语气/节奏，不要改成"AI 味"
4. **不碰 immutable**：改稿发生在 pre-predict 阶段（script.md 还没被 hash 锁定）。如 script 已进过 /cheat-predict，拒绝改稿——预测已锁定，改稿会让 hash 不匹配。提示用户走 `_redo.md` 路径

## 改稿日志（写到 brief/audit 末尾）

每条改动记录一行，便于复盘：

```markdown
## 改稿日志

| # | 位置 | 类型 | 问题来源 | 动作 | 状态 |
|---|---|---|---|---|---|
| 1 | 开头钩子（第 33-39 行） | 机械改动 | 钩子脱节 | "朝廷是空的" → "古人选官两千年，我两步搞定" | ✅ 已执行 |
| 2 | 第 3 段结论句 | 需创作者输入 | 推理缺失 | 插入 TODO 注释 | ⏳ 待用户补 |
| 3 | 第 5 段营销话术 | 机械改动 | 营销腔 | 删"你也能 3 天学会" | ❌ 用户拒绝 |
```

## 改稿后流程

```
Phase 5 改完
  ↓
情况 A：改了 ≥1 条 [机械改动]
  → 提示"改完了。建议重跑 /cheat-who-for（或 /cheat-open-source）看红旗是否解除，再 /cheat-predict"
  → 重跑会覆盖 brief/audit（保留改稿日志段）

情况 B：全是 [需创作者输入]，用户还没补
  → 提示"位置已标好 TODO 注释。你补完后说'补完了'，我帮你重跑审查 + 进 predict"

情况 C：用户全部跳过 / 拒绝
  → 提示"OK 不改。brief/audit 保留原状（red_flag: true）。你可以直接 /cheat-predict，但 predict 会读到 red_flag 降低 confidence"
```

## Refusals

- 「帮我全自动改完，不用逐条确认」 → 拒绝。改稿动的是创作者的稿子，必须逐条确认。可提供"批量看完全部建议后一次性 ok"的快路径，但执行前仍要展示每条 diff
- 「把 [需创作者输入] 的部分你也写了」 → 拒绝。推理过程/真实经历是你的剧本，AI 写的 = 编造，违反自我开源原则（cheat-open-source 尤其）
- 「script 已经 predict 过了，直接改 script」 → 拒绝。predict 锁定了 script hash，改稿破坏 immutability。改走 `scripts/<id>_redo.md` 新建稿子
